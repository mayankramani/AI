01

Why MCP exists

Every AI app needs the same things — files, databases, APIs, memory. Before MCP, every team rebuilt those integrations from scratch in incompatible ways. MCP is the USB-C port for AI: one connector, every device.

The four pain points MCP was built to fix

1
Fragmented integrations

Every AI client (Cursor, Claude Desktop, VS Code) wrote a custom connector for every tool. Same logic, rebuilt N times.

2
The N × M problem

N clients × M tools = N×M custom integrations to write and maintain. A 10×50 ecosystem means 500 bespoke connectors.

3
Inconsistent context plumbing

Each team invented its own prompt format, tool-call schema, and auth flow. Nothing was portable across apps.

4
Stateless tool binding

LangChain-style bind_tools() re-declares tools per session. No central registry, no shared lifecycle, no discoverability.

Before MCP vs With MCP — the integration math

🕸Before MCP — the N × M tangle

  • Each AI app implements its own GitHub connector, its own Slack connector, its own Postgres connector…
  • Tool authors must ship a different SDK for every client
  • Changing one API breaks dozens of bespoke clients
  • No standard for auth, schema discovery, or capability negotiation
  • Adding a new tool to your IDE means waiting for the IDE vendor to ship support
  • 10 clients × 50 tools = 500 hand-written integrations

🔌With MCP — the N + M plug-and-play

  • Each tool ships one MCP server, usable by every MCP-compatible client
  • Each AI app implements one MCP client, usable with every MCP server
  • 10 clients + 50 tools = 60 implementations — and they all interoperate
  • JSON-RPC 2.0 + capability negotiation = standard discovery
  • OAuth 2.0 (remote servers) = standard auth
  • Build once, adopted everywhere — like LSP did for IDEs and programming languages
The Core Insight
MCP = LSP for AI agents

The Language Server Protocol let one Python plugin work in VS Code, Vim, Emacs, JetBrains — instead of N editors × M languages. MCP does the exact same trick for AI: one Slack MCP server works in Claude Desktop, Cursor, Windsurf, Cline, Goose, and any future client — instead of every AI app maintaining its own Slack integration.

02

What is MCP, really?

Three words doing three jobs. Model is the LLM. Context is everything outside the model's training data. Protocol is the standardised JSON-RPC 2.0 conversation that bridges them. Released by Anthropic in November 2024 (designed by Mahesh Murag); fully implemented as Python and TypeScript SDKs; already supported by Cursor, Windsurf, Cline, Claude Desktop, Claude Code, and a growing list of agents.

The three words — unpacked

🤖

Model

the LLM at the heart

GPT, Claude, Llama, DeepSeek — whatever reasoning engine drives your app. Powerful, but boxed in by training-cutoff knowledge and a finite context window. Without external help it can only talk about the world it saw during training.

🧠

Context

everything outside the weights

Files, database rows, API responses, user preferences, conversation history, RAG retrievals. The extra information the model needs to answer correctly. Generative AI quality is bounded almost entirely by how good your context-management is.

📡

Protocol

JSON-RPC 2.0 over stdio / HTTP+SSE

The standardised conversation. Defines how a client lists tools, calls them, subscribes to changes, and negotiates capabilities — once, for the entire ecosystem. No more bespoke wire formats per integration.

Anthropic's one-liner
"A USB-C port for AI applications"

HTTP standardised how browsers talk to servers. LSP (Language Server Protocol) standardised how editors talk to language tooling. MCP standardises how AI applications talk to external data and tools. Same shape of solution — one wire format, infinite peripherals. MCP explicitly takes its design inspiration from LSP.

Why "context" is the whole game

Out of the box, every generative model is limited by its pre-training cutoff, its token window, and its conversation history. The way to push past those limits is to feed it the right context, on demand. MCP is a protocol for doing that feeding in a structured, standardised, secure way.

📝Where context comes from — by modality

  • Text models (GPT, Claude, Llama) — prompt, token window, chat history, RAG
  • Image / multimodal (DALL·E, Gemini) — text description + visual reference
  • Code models (Codex, DeepSeek-Coder) — surrounding code, language syntax, API docs
  • Speech / audio (Whisper, AudioPaLM) — prior audio segments, acoustic features

🔄Why agents make context dynamic

  • Agents don't get one fixed prompt — they decide at runtime what context they need
  • They search data sources, request specific records, chain follow-up calls
  • Each external system used to need its own bespoke connector
  • MCP is the standard wire those agents speak when they reach out

The fragmentation problem MCP retires

🕸Before MCP — the N×M tax

  • Every AI app wrote bespoke glue for every tool / data source
  • Each data source emitted messages in its own shape (or no protocol at all)
  • Inconsistent prompt logic + tool-calling conventions across teams & vendors
  • N client apps × M tools = N×M custom integrations to build and maintain
  • Switch the AI app — rewrite everything. Bump a tool's API — hunt every caller.

🔌After MCP — N + M plug-ins

  • Each AI app implements the MCP client side once
  • Each tool / data source ships an MCP server once
  • Any compliant client ↔ any compliant server, with zero extra glue
  • MCP becomes the fungibility layer between AI clients and capability providers
  • Open standard — not owned by any single AI vendor

What MCP standardises (in one breath)

📂

Sharing context with the model

via resources/*

Servers expose data — files, DB schemas, wiki pages, API responses — each addressable by URI. Hosts decide how to surface them: tree view, search, auto-include by heuristic. Application-driven.

🔧

Exposing tools to AI systems

via tools/*

Servers publish named functions with JSON-Schema. The LLM autonomously picks when to call which. Discovery (tools/list), invocation (tools/call), change notifications — all standardised. Model-controlled.

📝

Building composable workflows

via prompts/*

Servers ship pre-baked prompt templates (typically slash-commands) that users explicitly pick. Combined with composability (clients-as-servers), you can build agent hierarchies that delegate. User-controlled.

The four MCP roles (preview of the next section)

Host → Client → Server → Capability

Stateful sessions, not one-shot calls

🔄

Three message types (JSON-RPC 2.0)

  • Requests — bidirectional, expect a response (e.g. tools/call)
  • Responses — success result or structured error, matched by ID
  • Notifications — fire-and-forget, no response (e.g. notifications/tools/list_changed)
🤝

Capability negotiation up front

  • On initialize, client + server both declare what they support
  • Server: tools? resources (with subscribe / listChanged)? prompts?
  • Client: sampling? roots?
  • Each capability unlocks specific methods — nothing is invoked unless both sides agreed

Four design principles (worth memorising)

1️⃣

Servers should be trivial to build

One async function + decorator = a working server. Hosts carry the orchestration weight. Simple interfaces win over clever ones.

2️⃣

Servers should be highly composable

Each server owns one focused capability. Many small servers > one giant one. Shared protocol means they snap together with no custom glue.

3️⃣

Servers can't see the whole conversation

Hard isolation. A server gets only the slice it needs. Full conversation history stays in the host. Cross-server interactions are mediated, never direct.

4️⃣

Features can grow progressively

Core protocol is intentionally small. Optional capabilities are negotiated, not assumed. Clients and servers evolve independently; backwards compatibility is a feature.

Where MCP is heading — remote servers, registry, agents

🌐

Remote servers + OAuth 2.0

Beyond local stdio: servers hosted on public URLs over HTTP+SSE, secured with OAuth 2.0 handshake. Users authorise once; the server holds the token. Drops the barrier to entry to open a URL.

📒

MCP Registry (in development)

A central discoverable index of verified MCP servers — capabilities, versions, trust signals, official-vendor blessings. Solves the "how do I find the right server?" problem.

🧬

Self-evolving agents

Once a registry exists, an agent can encounter a task it can't do, search the registry, install the missing server, and proceed. Agents stop being limited to their pre-programmed tool set.

Bottom line
MCP is a standardised, stateful, capability-negotiated, security-isolated way to feed any AI app any context from any source.

The protocol is small on purpose. The leverage comes from the ecosystem: any client + any server + any model + any tool, all snapping together because they speak one language. The next section makes the roles concrete with a plain-English walkthrough.

03

MCP in plain English

Before any code or protocol detail — here is the whole idea in everyday language, for a reader who has never seen MCP. Four roles, one shared language, a few concrete examples. If you remember only this section, you'll still understand the rest.

The 30-second restaurant analogy

🍽You sit at a table (the Host)

  • The Host is the AI app you actually use — Claude Desktop, Cursor, Windsurf, ChatGPT Desktop, an IDE plugin
  • It owns the conversation, the screen, the user's trust
  • It decides which kitchens (servers) it's allowed to talk to

🧑‍🍳A waiter walks to each kitchen (a Client)

  • The Client is a tiny connector the Host spawns — one per kitchen
  • It speaks the shared protocol (MCP / JSON-RPC) on your behalf
  • One client ↔ one server. Strict 1-to-1. Servers can't talk to each other

🏫The kitchen prepares the dish (a Server)

  • An MCP Server is a small program wrapping one capability — your filesystem, GitHub, Slack, Postgres, Jira, a search API
  • It exposes a menu of things it can do, in a standard format
  • You can write one in ~30 lines of Python or TypeScript

📜The menu items are the Tools, Resources, Prompts

  • Tools — actions the AI can run ("send_email", "query_db")
  • Resources — data the AI can read (a file, a row, a wiki page)
  • Prompts — templates the user picks (slash-commands)
The one-line definition
MCP is a standard way for an AI app to ask outside programs for context and to invoke actions — without bespoke integration code for every combination.

Anthropic released it in November 2024 (designed by Mahesh Murag). Think of it as USB-C for AI: one socket, many devices. Before MCP, every AI app needed custom glue code for every tool — the famous N×M problem. After MCP, build a server once, and any MCP-aware app can use it.

One picture — the four roles together

Host · Clients · Servers · Capabilities

The four roles, defined precisely

app-controlled
🏠

1. Host — the AI application

Claude Desktop · Cursor · Cline · Windsurf · Goose
  • The program the human actually opens
  • Owns the LLM (or calls an external provider)
  • Spawns Clients, enforces consent + security policy
  • Keeps the full conversation — servers never see it
  • Aggregates context from many Clients into one prompt
app-controlled
🔌

2. Client — the per-server connector

one Client = one stateful session = one Server
  • Created and owned by the Host (1:1 with a Server)
  • Performs the initialize handshake + capability negotiation
  • Routes JSON-RPC messages both directions
  • Subscribes to notifications (resource changed, tool list changed)
  • Hard isolation — a client never sees another server's data
server-controlled
🏭

3. Server — the capability provider

one server wraps one system (filesystem, GitHub, Postgres, Jira…)
  • Small, focused process — "do one thing well"
  • Declares which primitives it offers (tools / resources / prompts)
  • Runs locally over stdio or remotely over HTTP+SSE
  • Can't see the conversation, can't see other servers
  • Build once — every MCP-aware Host gets to use it
model-controlled
🔧

4. Tools — the LLM's actions

named function + JSON schema + description
  • What the language model can do, not just read
  • Server publishes them via tools/list; LLM picks via tools/call
  • Each call returns structured content (text / image / JSON)
  • Host always shows the user a consent prompt before execution
  • Tools ride alongside Resources (read-only data) and Prompts (user-triggered templates)

Walking through one real interaction

1

You launch Claude Desktop — the Host boots up

Claude Desktop reads its config file, sees you registered a filesystem MCP server and a GitHub MCP server. It spawns two Clients, one per server, and each runs an initialize handshake.

2

The Clients ask each Server "what can you do?"

Each Server answers with a list of capabilities + JSON schemas. Claude (the LLM) now knows that read_file, list_dir, list_issues, create_pr all exist — along with the exact shape of every argument.

3

You ask: "Summarise the open issues in my repo and draft a release note"

The LLM reasons: "I need list_issues from the GitHub server, then I'll write text". It emits a tools/call request through the GitHub Client.

4

The Host pops a consent UI

You see exactly which tool will run, with which arguments. You click Approve. The Client forwards the JSON-RPC call to the Server.

5

Server executes, returns structured result

The GitHub server hits the GitHub API, packs the issues into a tool result, sends it back. The LLM receives the data, writes the release note, and you see the final answer — with the tool-use trace visible.

What the standard solves — the N×M problem

🕸Before MCP — N×M bespoke integrations

  • N AI apps × M tools = N×M custom adapters
  • Every team reinvents prompt logic + tool-calling glue
  • Switch the AI app? Rewrite every integration
  • Switch the tool's API version? Hunt down every caller

🔌With MCP — N + M plug-ins

  • Each AI app implements the MCP client side once
  • Each tool ships an MCP server once
  • Any client ↔ any server, instantly, with no extra glue
  • Mental model borrowed from the Language Server Protocol (LSP) that fixed editor ↔ language integration
Who benefits
App builders ship faster · tool vendors get free distribution · users get richer AI

App developers drop in any MCP server with zero per-integration work. Tool vendors (Shopify, Grafana, Sentry…) build their server once and instantly reach every MCP-compatible client. Enterprises separate concerns cleanly — one team owns the vector-DB MCP server, every other team just consumes it. End users get AI apps that actually know things about their data and can take real actions — not just chat.

MCP vs RAG vs Tool Binding

People ask: "Isn't this just RAG?" No. RAG fetches text. Tool binding wires one agent to one tool. MCP is the wiring standard that lets any agent talk to any tool or data source without bespoke code.

Evolution of LLM ↔ tool integration

RAG vs MCP — overlap and difference

📚Traditional RAG (without MCP)

  • App manually owns the vector store, embedder, retriever
  • Adding a new document source = rewriting app code
  • Each integration is bespoke and tightly coupled
  • Read-only retrieval — fetches text, can't take actions
  • One-shot per query, no bidirectional protocol
  • Hard to swap implementations without breaking the app

🔌MCP (with optional RAG inside a server)

  • App asks: "give me context about X" — server decides how
  • Swap a document source by swapping the server, app untouched
  • Standard Resources primitive — same shape for any data
  • Tools primitive enables actions (write file, send Slack, run query)
  • Bidirectional + stateful — supports memory, subscriptions, sampling
  • A RAG pipeline is one valid implementation behind an MCP server

05

Architecture — Host · Client · Server

MCP uses a three-role topology with strong isolation. One host process manages many clients, each holding a 1-to-1 session with a single server. The host is the trust boundary.

Host · Client · Server — one host, many isolated sessions

🏠

Host

the trust boundary
  • Creates and supervises all clients
  • Holds the LLM + the full conversation history
  • Enforces user consent before any tool call
  • Aggregates context from multiple servers
  • Coordinates sampling requests back to the LLM
🔗

Client

one per server, isolated
  • 1-to-1 stateful session with exactly one server
  • Handles protocol negotiation + capability exchange
  • Routes JSON-RPC messages bidirectionally
  • Manages subscriptions and notifications
  • Never sees other servers or full chat history
🛠

Server

focused capability provider
  • Exposes Tools, Resources, Prompts via MCP primitives
  • Local subprocess (stdio) or remote service (HTTP+SSE)
  • One focused responsibility (files, GitHub, Slack, DB…)
  • Can be built by anyone — official, community, or in-house
  • Reusable across every MCP-compatible client
Why this topology
Servers can't read the conversation, nor see each other

The host hands each server only what it needs. A GitHub server never sees your Slack messages; a Postgres server never sees your filesystem reads. This isolation is a core MCP design principle — and the reason MCP is safe to install from untrusted sources, while raw "give the LLM API keys to everything" agent designs are not.

06

The three server primitives

Every MCP server exposes some combination of Tools, Resources, and Prompts. Each has a different controller — that's the key distinction.

model-controlled
🔧

Tools

executable functions the LLM can call

Side-effectful operations: query a DB, send a message, create a file, hit an API. Each tool has a name, a JSON-schema input, and a description. The LLM decides when to call them based on context.

  • Discovered via tools/list
  • Invoked via tools/call
  • Capability flag: tools.listChanged
  • Examples: create_issue, send_email, run_query
application-controlled
📂

Resources

read-only data with stable URIs

Files, DB schemas, log streams — anything addressable. The host application decides how to surface them (auto-include, user-pick, search). Each has a unique URI like file:///x or postgres://t/users.

  • Listed via resources/list
  • Read via resources/read
  • Optional subscribe for change notifications
  • Examples: project files, schemas, dashboards, log tails
user-controlled
💬

Prompts

parametrised prompt templates

Pre-built instructions the user can pick — typically surfaced as slash commands. The user (not the LLM) decides when to invoke. The server fills in the arguments and returns a structured message sequence.

  • Listed via prompts/list
  • Retrieved via prompts/get
  • Often surfaced as /slash-commands in the UI
  • Examples: /explain-code, /summarise-pr, /triage-bug
The Controller Matters
Who decides when to invoke — model, app, or user?

This single axis determines security, UX, and observability. Tools are model-controlled (so the host must enforce user consent). Resources are app-controlled (so the host UI shows them as a tree/list/auto-include). Prompts are user-controlled (so the user explicitly fires them). Mixing these up is the #1 design mistake when building a server.

07

Client features — Roots & Sampling

Servers aren't the only ones with capabilities. Clients can expose Roots (filesystem boundaries) and Sampling (LLM access on behalf of the server) — the keys to advanced, agentic behaviour.

📁

Roots

file:// URIs the server may operate within

Clients tell servers "you can touch these directories, nothing else." A code-editor host might expose the open project folder; a desktop client might expose ~/Documents. Servers receive notifications when the root list changes.

  • Listed via roots/list
  • Each root: { uri: "file:///path", name?: "Project" }
  • Notification: roots/list_changed
  • Hard boundary — server cannot escape it
🧬

Sampling

server asks client to run the LLM

A server can request "please run this completion through your LLM and give me the result." The client controls model choice, approves the prompt, and decides what the server gets to see. This is what enables nested, agentic behaviour without giving servers API keys.

  • Request: sampling/createMessage
  • Client approves the prompt and the model
  • Server has zero credentials — host pays for tokens
  • Enables agents to call other agents recursively
🧩

Composability

any node can be both client and server

Client/server is a logical role, not a physical one. An orchestrator agent acts as a host to a research sub-agent (server) — and that sub-agent in turn acts as a host to a web-search server. This is how multi-agent systems compose without bespoke wiring.

  • Layered agent architectures
  • Each layer = its own trust boundary
  • Reuse third-party agents as MCP servers
  • Specialise: orchestrator → researcher → fact-checker
08

Lifecycle & JSON-RPC messages

Every MCP session goes through three phases — Initialize, Operate, Shutdown — and exchanges three message shapes: Request, Response, Notification.

Session lifecycle — handshake, normal operation, clean shutdown

The three JSON-RPC 2.0 message shapes

↩️

Request

bidirectional · expects a response
// client → server { "jsonrpc": "2.0", "id": 42, "method": "tools/call", "params": { "name": "get_docs", "arguments": { "query": "Chroma DB", "library": "langchain" } } }
✅

Response

result or error · matches request id
// server → client { "jsonrpc": "2.0", "id": 42, "result": { "content": [ { "type": "text", "text": "Chroma is a vector DB..." } ], "isError": false } }
📢

Notification

one-way · no id · no response
// either direction { "jsonrpc": "2.0", "method": "notifications/tools/list_changed" } // client tells server it is ready { "jsonrpc": "2.0", "method": "notifications/initialized" }
Capability Negotiation
Each side declares what it supports — once, upfront

During initialize, the client declares features like sampling and roots; the server declares tools, resources, prompts and sub-flags like resources.subscribe. The session may only use what both sides agreed on. This is why MCP evolves cleanly: a newer client can ignore unknown capabilities; an older server simply never advertises features it doesn't have.

Real example — Claude Desktop + Filesystem MCP

The fastest way to feel MCP: drop the pre-built filesystem server into Claude Desktop and watch it read, write, and search your real files. Zero code.

1

Install Claude Desktop & Node.js

Grab Claude Desktop (macOS or Windows) and confirm Node is on PATH — the official filesystem server runs via npx.

$ node --version v20.11.0
2

Enable Developer Mode

Top-left Claude menu → Help → Enable Developer Mode. A new Developer tab appears in Settings.

3

Edit claude_desktop_config.json

Settings → Developer → Edit Config. Paste the JSON below, replace username with yours, and pick folders Claude is allowed to touch.

{ "mcpServers": { "filesystem": { "command": "npx", "args": [ "-y", "@modelcontextprotocol/server-filesystem", "C:\\Users\\username\\Desktop", "C:\\Users\\username\\Downloads" ] } } }
4

Restart Claude Desktop

Claude relaunches the server as a child process and talks to it via stdio. You'll see a small 🔨 (hammer) icon next to the input box — that means the server registered its tools.

tools/list → [ read_file, write_file, move_file, search_files, list_directory, ... ]
5

Try it

Ask Claude in plain English. It picks the right tool, asks for your approval, then executes.

you: Write a haiku about coffee and save it to my Desktop as coffee.txt claude: [tool: write_file] → approve? ✓ claude: Done. Saved coffee.txt to ~/Desktop. you: Move every .png on my Desktop to a new folder "Images" claude: [tool: list_directory + move_file ×N] → approve? ✓
What just happened
No glue code. Standard protocol. Real actions.

You added a fully-functional file-management capability to a desktop LLM by editing one JSON file. The same server works in Cursor, Cline, and any other MCP-compatible client without modification. This is the "build once, adopt everywhere" promise made concrete.

Composability & self-evolving agents

The two features that turn MCP from "tool plumbing" into something genuinely new: composable agents and the upcoming MCP registry.

An orchestrator agent that is itself a server — and discovers new servers at runtime

🧩Composability

  • Client/server is a logical role — any node can be both
  • An orchestrator agent looks like a server to its caller and a host to its sub-agents

  • Specialised agents (research, coding, fact-checking) compose like Unix pipes

WebMCP — MCP for the browser

The W3C just standardised an MCP-shaped API inside the browser itself. No server process, no JSON-RPC wire, no Python — websites declare callable tools directly to AI agents through navigator.modelContext. Released as a W3C Draft Community Group Report on 2026-02-10; live behind a flag in Chrome 146 Canary.

The problem WebMCP solves — agents reverse-engineering UIs

📸Screenshot approach (Claude for Chrome, etc.)

  • Agent captures the page as an image, ships it to a vision model
  • Vision model guesses where the buttons / fields are by pixels
  • Every CSS animation, modal, or layout shift breaks the agent
  • Huge token cost per turn + multi-second latency
  • A 3-step task balloons into dozens of capture→infer→click loops

🕸DOM parsing approach

  • Agent slurps raw HTML, hunts for forms / buttons / links
  • SPAs emit thousands of nested nodes that exist purely for layout
  • Class names like .css-1a2b3c tell the model nothing
  • DOM was designed to describe presentation, not capabilities
  • Pattern-matching burns tokens on plumbing markup
The conceptual inversion
Stop making the agent guess. Have the website declare its tools.

Both legacy approaches force the AI to reverse-engineer capabilities from an interface designed for human eyes. WebMCP flips it: the page tells the agent "here are the things I can do, here are the parameters I need, here is how to invoke them." Same inversion of control MCP did for servers — now in the browser.

Dual-layer web — human pixels above, machine schemas below

Two APIs — Declarative for forms, Imperative for everything else

declarative
📋

HTML form attributes

three new attrs on <form>

An HTML form is already a structured tool — it has an action, a method, and typed inputs. Three new attributes make it explicit to agents.

  • toolname — required, e.g. "searchFlights"
  • tooldescription — required, natural language
  • toolautosubmit — optional, keeps human in the loop by default
  • Server differentiates via SubmitEvent.agentInvoked
<form toolname="searchFlights" tooldescription="Search for flights between two airports."> <input name="origin" pattern="[A-Z]{3}" required> <input name="destination" pattern="[A-Z]{3}" required> <input name="date" type="date" required> <input name="passengers" type="number" min=1 max=9> <button type="submit">Search Flights</button> </form>
imperative
⚙️

navigator.modelContext.registerTool()

JS API for dynamic workflows

For SPAs and anything that isn't a single form submission. The execute handler is just JavaScript — it calls the same functions the visual UI does. No separate API layer.

  • Available on navigator.modelContext (HTTPS only)
  • registerTool() adds one tool at a time
  • provideContext() swaps the whole tool set (e.g. after sign-in)
  • Tools persist for the lifetime of the page (navigation clears them)
if ('modelContext' in navigator) { navigator.modelContext.registerTool({ name: "addToCart", description: "Add a product to the cart by SKU.", inputSchema: { type: "object", properties: { sku: { type: "string" }, quantity: { type: "integer", minimum: 1, maximum: 99 } }, required: ["sku", "quantity"] }, execute: async (params) => { const product = await fetchProduct(params.sku); const r = await addItemToCart(product.id, params.quantity); return { type: "text", text: JSON.stringify({ success: true, cartTotal: r.total }) }; } }); }

End-to-end interaction — sequence diagram

User · Agent · Browser · Page — one structured call replaces dozens of screenshot loops

Security model — two trust boundaries, one user consent loop

🛡

Boundary 1 — registration

site → browser → agent

The page exposes its tool surface. Protected by web platform defaults: HTTPS-only secure context, same-origin policy, and Content Security Policy still apply.

  • HTTPS required — no http:// or insecure contexts
  • Tools inherit the origin's security boundary
  • CSP enforcement carries through to tool registration
🔐

Boundary 2 — invocation

agent → browser → page (bidirectional)

The high-risk step. Untrusted agent input flows in; potentially sensitive results flow out. The browser inserts a human-in-the-loop consent prompt per invocation.

  • Per-call consent — user explicitly approves each tool execution
  • SubmitEvent.agentInvoked lets servers tell apart human vs agent
  • Read-only hints let safe query tools skip the prompt
  • Tools inherit the user's existing session — no OAuth between agent and site needed
The deeper design insight
Registering tools is the AI safety policy

When you call registerTool('search_restaurants', …) you are saying "this action is intentionally available to AI agents." If you never register cancel_booking, no agent can ever cancel a booking on your site — not because of a runtime block, but because the surface doesn't exist. Compare this to the screenshot/DOM era where AI safety was an after-the-fact detection problem. WebMCP makes it a deliberate, auditable design decision upfront. The "deadly triad" (one agent with simultaneous access to banking, email, and social tabs) is exactly why per-invocation consent + domain-level isolation are non-negotiable in the spec.

Known limitations (Feb 2026)

Tool discovery requires navigation

No way yet to learn a site's tool surface without visiting it. A .well-known/webmcp manifest is on the roadmap.

Client-side only

WebMCP tools live in frontend JavaScript. For direct backend access, Anthropic's classic MCP is still the right protocol.

Single-tab scope

Tools persist only for the lifetime of one page. Navigation wipes them. No cross-tab state via the API.

Evolving API surface

Draft CG Report, not a finalised standard. Expect breaking changes through 2026.

Browser support narrow

Chrome Canary only, behind a flag. Edge expected. No Firefox / Safari commitment yet.

Polyfill required for breadth

For now, ship MCP-B as a polyfill if you need WebMCP behaviour outside Canary.

The bigger picture
A second, machine-native interface layer for the web

For 30 years the web had exactly one interface — pixels rendered for human eyes. WebMCP introduces a second: a structured, schema-driven layer agents consume natively. Anand Sagar (Google) calls it "the USB-C of AI agent interactions with the web" — one standardised socket replacing the current tangle of bespoke scrapers and click-bots. Adoption depends on developers shipping tools, vendors beyond Chrome implementing the spec, and the W3C process completing — but the trajectory is clear.

Security & trust

MCP servers execute arbitrary code on your behalf and can read whatever you point them at. The protocol bakes in four trust principles that every implementor must respect.

1
User consent & control

Every data access and every tool invocation requires explicit user approval. The host UI must surface what is about to happen, in plain language.

2
Data privacy

The host never transmits resource data to other servers (or anywhere else) without user consent. Servers see only the slice of context they need.

3
Tool safety

Tools = arbitrary code execution. Treat unknown servers like running random scripts off the internet — because that's exactly what they are.

4
Sampling controls

When a server requests an LLM call, the user/client approves the prompt, picks the model, and decides what the server gets to see back.

Practical rules
Three commandments before installing any MCP server

1. Read the source (or trust the publisher). The official @modelcontextprotocol/* servers and verified vendor servers (Shopify, Grafana, GitHub) are safe defaults. 2. Scope filesystem roots tightly — don't hand a random server / or C:\. 3. Use OAuth 2.0 for remote servers; never paste long-lived API keys into a server config you didn't write.

13

Production pitfalls

What breaks first when an MCP server goes from "works on my desktop" to "serves an org."

Tool description rot

The LLM picks tools by name + description. Vague descriptions → wrong tool gets called → silent quality loss.

Fix: write USE FOR / DO NOT USE FOR sections; include 3–4 example queries; test routing with real prompts.

Returning too much text

A scraper that returns 50 KB per call burns the context window in three tool uses. The LLM then truncates the conversation.

Fix: truncate to ~4–8 KB per tool result; paginate; return summaries with optional "fetch full" follow-up.

No timeouts on remote calls

A slow upstream API blocks the whole tool call, the client gives up, the user sees "tool failed" with no detail.

Fix: wrap every HTTP call in httpx timeouts; return structured error objects with isError: true.

Secrets in config files

Pasting OPENAI_API_KEY=sk-... into claude_desktop_config.json puts it in a world-readable file forever.

Fix: load secrets via .env + python-dotenv; for remote servers use OAuth 2.0; never commit configs.

Forgetting capability negotiation

Calling resources/subscribe on a server that didn't declare resources.subscribe just errors out — but only at runtime.

Fix: always check the initialize response capabilities map before calling optional features.

Servers that read the whole conversation

Resist the temptation to slurp chat history into the server. It breaks isolation, leaks data, and violates the design principle.

Fix: the host curates context. If you need state, store it under a server-owned URI and expose via Resources.

One mega-server with 50 tools

The LLM has to scan every tool description on every turn. Routing degrades fast past ~15–20 tools.

Fix: split by domain — one server per logical capability area. Composability is free; tool sprawl is not.

No version pinning

npx -y @modelcontextprotocol/server-filesystem fetches latest on every launch. A broken release silently breaks your workflow.

Fix: pin a version (@1.2.3) in production configs; bump deliberately.

What's next — the 2026-07-28 RC

The next MCP spec release candidate lands on 2026-07-28. The headline: MCP becomes stateless at the protocol layer. The deeper story is that every part of an agent workflow — state, capabilities, authorisation, observability — gets pushed into explicit, visible primitives. Plumbing made auditable.

Before RC vs After RC — where state lives

per request

Stateless protocol — but state doesn't disappear, it becomes explicit

🫥Old shape — hidden in transport metadata

  • Server returned Mcp-Session-Id; client had to carry it
  • Sticky-session load balancers, shared session stores
  • Workflow state invisible to the model — pure plumbing
  • Gateways had to understand MCP to route correctly

🪟New shape — visible handles in the request body

  • Server returns explicit handles: basket_id, browser_id, repo_id
  • Model passes them back in subsequent tools/call invocations
  • State is now part of agent reasoning — and the audit log
  • Tool authors own scoping / validation / expiry of each handle
// step 1 — server returns a handle { "basket_id": "bkt_123" } // step 2 — model passes it back, any server instance can serve this { "basket_id": "bkt_123", "item_id": "sku_456" }

Extensions — a real lifecycle for capabilities that aren't core

📦

Extensions framework

reverse-DNS · own repo · own version

Extensions get IDs, dedicated ext-* repos, delegated maintainers, and an extensions map negotiated during initialize. Core stays small; the ecosystem still experiments.

  • Client + server advertise extension support during init
  • Both sides versioned independently from the main spec
  • Connection still works if an extension isn't supported on one side
🖼

MCP Apps

first official extension

Servers can ship interactive HTML interfaces the host renders in a sandboxed iframe. UI templates are declared up-front so hosts can prefetch, cache, and security-review them.

  • Button clicks in the app route back through MCP JSON-RPC
  • Same audit + consent path as a direct tool call
  • Useful for rich data displays, configurators, dashboards
⏳

Tasks

second official extension

Long-running work without polling tricks. A tools/call can return a task handle; the client drives it with tasks/get, tasks/update, tasks/cancel.

  • Server-directed — server decides when a call becomes a task
  • tasks/list intentionally removed (no sessions = no safe scoping)
  • Cleaner than per-server polling protocols

Deprecations — MCP narrows its responsibilities

Deprecated featureWhyNew direction
RootsWorkspace scoping bled into the protocolTool parameters · resource URIs · server config
SamplingModel access overlapped with provider APIsDirect LLM provider integration
LoggingDistributed-systems observability is a solved problemstderr for stdio · OpenTelemetry for structured traces
Deprecation policy (new in this RC)
12 months minimum between deprecation and removal

Methods + capability flags are annotation-only for now. They still work in this release and in every spec version published within a year of it. Removal requires a separate SEP. Translation: nothing breaks tomorrow — but start planning the migration. Distributed-systems observability belongs with OpenTelemetry; model selection belongs with provider APIs; workspace scope belongs in tool inputs. MCP is keeping only the client-server contract it owns best.

Other RC changes worth flagging

🔑

Authorization tightening

OAuth / OIDC sharper rules
  • Clients must validate which issuer sent the auth response
  • Credentials stay bound to the AS that issued them
  • Token-refresh semantics clarified for multi-server clients
  • Desktop / CLI clients get explicit localhost-redirect support
📐

JSON Schema 2020-12 (full)

tool schemas can finally describe reality
  • Inputs keep type: "object" root but allow composition + conditionals + $ref
  • Output schemas are unrestricted; structuredContent any JSON
  • Implementers warned: cap validation work, don't blindly follow external $ref
🏛

Governance + conformance

SEPs need matching scenarios
  • Feature lifecycle: Active → Deprecated → Removed with min 12-month gap
  • Standards-Track SEPs must ship conformance tests to reach Final
  • Smaller implementations have a realistic shot at compatibility
  • Less risk that the protocol drifts toward one vendor's behaviour

Migration checklist for the 10-week window

1

Probe statelessness — does your server need sticky sessions?

Send tools/call across instances behind a round-robin LB. Anywhere it breaks is a hidden session dependency to convert to an explicit handle.

2

Inventory every piece of cross-call state

Repo paths, browser sessions, deployment envs, task IDs. Each one becomes a scoped, validated, expirable handle visible to the client.

3

Plan migration off Roots, Sampling, Logging

Annotation-only today but on the clock. Move workspace scope into tool inputs, model calls to provider APIs, telemetry to OpenTelemetry.

4

Audit authorisation flows

Issuer validation, credential binding, token refresh, localhost redirects. Multi-server clients especially — weak OAuth semantics become security incidents at scale.

5

Tighten tool schemas — but cap their complexity

Lean on full JSON Schema 2020-12 for honest input contracts. Don't recurse external $ref blindly; bound validation work.

6

Feed weird cases back to the spec repo

Open protocols get stronger when implementers surface real-world friction before the spec hardens. The spec repo + Working Group Discord are the right channels.

The strategic read
MCP is choosing operational clarity over feature creep

Every change in this RC trades convenience for visibility. Stateless protocol over hidden sessions. Explicit handles over magic IDs. Extensions framework over kitchen-sink core. Deprecations of Roots/Sampling/Logging because other systems already own those concerns better. Conformance tests over interpretation. A protocol that owns the right things and lets the ecosystem grow around it — that's the difference between a standard and a branded integration format.
