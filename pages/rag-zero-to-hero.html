01

Why RAG exists

An LLM on its own is a brilliant student who closed every textbook before the exam. It speaks fluently — but only about the world it saw during training. RAG hands the student the right page at the right moment.

The four failure modes RAG was built to fix

1
Stale knowledge

An LLM trained in 2023 can't answer "Who won yesterday's match?" — its world ends at the training cutoff.

2
Confident hallucination

Asked about a topic outside its data, the model fabricates plausible-sounding but invented facts.

3
No private knowledge

Your company's wiki, contracts, code repo, and customer history were never part of pre-training.

4
No citations

Even when correct, the model can't point to a source — making it unusable for legal, medical, or regulated work.

LLM-only vs LLM + RAG — side by side

🧠LLM alone (parametric memory only)

  • Answers from frozen training weights
  • No access to new or private data
  • Hallucinates when uncertain
  • Updating knowledge requires retraining or fine-tuning
  • Cannot cite specific sources
  • Cost of "knowing more" = cost of retraining

🔗LLM + RAG (parametric + retrieval)

  • Pulls fresh, relevant chunks from a vector store at query time
  • Indexes any private corpus — wiki, PDFs, tickets, code
  • Grounded answers — model says "I don't know" when retrieval is empty
  • Updating knowledge = re-indexing a document (seconds)
  • Every answer carries its source citations
  • Cost of "knowing more" = cost of one embedding call
The Core Insight
RAG = Search + Generation, glued by embeddings

The LLM stays small and frozen. The knowledge sits in a searchable vector index. At query time we retrieve the most relevant chunks and augment the prompt so the LLM generates a grounded answer. That's it — and that simple pattern powers ChatGPT's browsing mode, GitHub Copilot Chat, Notion AI, Perplexity, and every enterprise "chat with your docs" tool you've ever used.

The Simple RAG Pipeline

Two flows live in every RAG system: an offline ingestion pipeline that turns your documents into searchable vectors, and an online query pipeline that runs per request. They meet at the vector store.

End-to-end RAG flow — ingestion (top) meets query (bottom) at the Vector DB

Stage-by-stage breakdown

📄

1 · Load

Pull source documents from S3, SharePoint, Confluence, Git. Strip boilerplate, preserve structure (headings, tables, code blocks).

Tools: LangChain loaders, LlamaParse, Unstructured.io, PyMuPDF
✂

2 · Chunk

Break long documents into passages small enough to embed (200–800 tokens) but large enough to carry meaning.

Why: embeddings have a ~512 token ceiling on most models
🧮

3 · Embed

Convert each chunk into a dense vector (e.g. 384 or 1536 floats) that captures its semantic meaning.

Models: all-MiniLM-L6-v2, BGE-large, OpenAI text-embedding-3
🗄

4 · Store

Persist vectors + original text in a vector DB with an ANN index (HNSW, IVF) for sub-100 ms search.

Tip: always store original text — embeddings aren't reversible
🧮

5–6 · Embed Query

At query time, embed the user's question with the same model used for ingestion. Mismatched models = garbage results.

Latency budget: 20–80 ms for local models
🔍

7 · Retrieve top-K

Find the K chunks whose vectors are closest to the query vector (cosine similarity or dot product).

Typical K: 3 (chat) — 20 (with reranker)
📝

8a · Augment

Glue the retrieved chunks into a prompt template:
Context: {chunks}\nQuestion: {q}

Watch: total prompt must fit the LLM context window
🤖

8b · Generate

The LLM reads the context and writes an answer grounded in those chunks — with citations.

Prompt rule: instruct "answer only from context, else say I don't know"

Embeddings & Tokenization

Before any model can "understand" text, that text must become numbers. Tokenization breaks words into pieces; embeddings turn those pieces into vectors in a geometric space where meaning becomes distance.

"Mitochondria" → 4 BERT subword tokens → token IDs → vectors → pooled sentence embedding

Tokens are the atoms

"BPE", "WordPiece" and "SentencePiece" all do the same job: shatter text into reusable pieces.

  • Why subwords? A 30k vocabulary covers any English word. Rare words like "Mitochondria" decompose into known pieces.
  • The ## prefix in BERT marks "this token continues the previous one" — preserves the word boundary.
  • ~4 chars ≈ 1 token for English. Code, emoji, and CJK are denser.
  • Same tokenizer everywhere: the tokenizer used to embed your docs must match the one used at query time.
Real impact
Token count = cost & latency

GPT-4 charges per token. A 500-page PDF that tokenizes to 250 k tokens costs ~$2.50 just to embed once with OpenAI's ada-002. Pick a local model (all-MiniLM) for high-volume ingestion.

# Tokenize and embed with sentence-transformers
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("all-MiniLM-L6-v2")
tokens = model.tokenizer.tokenize("Mitochondria produce energy")
# → ['mit', '##och', '##ond', '##ria', 'produce', 'energy']

vector = model.encode("Mitochondria produce energy")
print(vector.shape)  # → (384,)  one 384-dim vector per sentence
Words with similar meaning cluster together. Distance = (1 − cosine similarity).

Meaning becomes geometry

An embedding model maps text into a high-dim space where related concepts sit near each other.

  • Cosine similarity — angle between vectors. 1.0 = identical, 0 = unrelated, −1 = opposite. Standard for normalized text embeddings.
  • Dimensionality — 384 (MiniLM) → 768 (BERT) → 1536 (ada-002) → 3072 (text-embedding-3-large). More dims ≠ always better; it's about training data quality.
  • The famous arithmetic: king − man + woman ≈ queen — early Word2Vec showed embeddings encode relationships, not just meaning.
  • Search = nearest-neighbor: embed the query, find the K closest vectors in your store. ANN indexes (HNSW, IVF) make this O(log N).
Why this matters for RAG
Semantic recall beats keyword search

A query "powerhouse of the cell" retrieves a chunk that only says "mitochondria generate ATP" — no word overlap, but the vectors sit ~0.1 apart in embedding space.

# Cosine similarity in practice
from sklearn.metrics.pairwise import cosine_similarity

q  = model.encode("powerhouse of the cell")
d1 = model.encode("Mitochondria generate ATP")
d2 = model.encode("The sun rises in the east")

cosine_similarity([q], [d1])  # → 0.89  highly related
cosine_similarity([q], [d2])  # → 0.04  unrelated
Each generation captured richer context. Modern RAG defaults to sentence-level models trained with contrastive loss.

Pick the right embedding model

Cost, latency, accuracy, and language coverage are the four knobs.

  • all-MiniLM-L6-v2 — 384 dim, 80 MB, runs on CPU. Free, fast, good enough for 80% of cases. Start here.
  • BGE-large / E5-large — 1024 dim, top of the MTEB leaderboard. Best quality you can self-host.
  • OpenAI text-embedding-3-small — $0.02 per 1M tokens. Production-ready, multilingual.
  • Voyage / Cohere / Mistral-Embed — domain-tuned variants (code, legal, multilingual).
  • Golden rule: never mix models. Re-embed the entire corpus if you change.
# Compare models on YOUR data, not just on benchmarks
from sentence_transformers import SentenceTransformer

candidates = [
    "all-MiniLM-L6-v2",      # 384d, fast, free
    "BAAI/bge-large-en-v1.5",  # 1024d, SOTA open
    "intfloat/e5-large-v2",    # 1024d, strong on QA
]
for name in candidates:
    m = SentenceTransformer(name)
    score = evaluate_recall_at_k(m, your_test_set, k=5)
    print(name, score)

Chunking Strategies

Chunking is the silent killer of RAG quality. Too small → no context. Too large → lossy embeddings. Wrong boundaries → answers cut in half. Four strategies cover 95% of real systems.

Fixed-size chunking

"Just cut every 500 tokens" — the simplest possible strategy and the right starting point.

Use when
  • Prototyping a new RAG pipeline
  • Documents are uniform (logs, tweets)
  • You need predictable throughput
Avoid when
  • Documents have clear structure (headings, tables)
  • Answers span paragraph boundaries
  • You need high recall on long Q&A
Real-world
Slack message archive search

Short uniform messages, no internal structure → 256-token fixed chunks with 32-token overlap works fine and runs at 50k docs/min.

def chunk_by_tokens(text, size=500, overlap=50):
    tokens = tokenizer.tokenize(text)
    chunks = []
    for i in range(0, len(tokens), size - overlap):
        chunk = tokens[i : i + size]
        chunks.append(tokenizer.decode(chunk))
    return chunks

Semantic chunking

Embed each sentence, drop a boundary wherever similarity collapses.

Use when
  • Each chunk needs to be coherent
  • Topic shifts are frequent
  • Quality > ingestion speed
Avoid when
  • You're ingesting millions of docs (slow)
  • Documents are short (< 1000 tokens)
Real-world
Customer support knowledge base

Long articles cover multiple problems. Semantic chunking keeps each "problem + solution" pair intact in one chunk → retrieval lands on a complete answer, not a fragment.

def semantic_chunk(sentences, threshold=0.5):
    embs = model.encode(sentences)
    chunks, cur = [], [sentences[0]]
    for i in range(1, len(sentences)):
        sim = cosine(embs[i-1], embs[i])
        if sim < threshold:           # topic shift detected
            chunks.append(" ".join(cur)); cur = []
        cur.append(sentences[i])
    chunks.append(" ".join(cur))
    return chunks

Hierarchical chunking

Store summaries at every level — retrieve coarse-to-fine for huge documents.

Use when
  • Documents are 100+ pages (manuals, books, contracts)
  • Section structure is meaningful
  • You want explainable retrieval ("from Ch 3 §2")
Avoid when
  • Short flat documents
  • Storage cost is constrained (3× more vectors)
Real-world
Aircraft maintenance manuals

A Boeing 787 maintenance manual is ~10,000 pages. Hierarchical chunking matches a query to the right chapter summary first, then surfaces 3-4 procedure paragraphs — vs scanning 50k flat chunks.

def hierarchical_chunk(doc):
    return {
        "doc_summary": summarize(doc),
        "sections": [
            {"summary": summarize(sec),
             "paragraphs": sec.split_paragraphs()}
            for sec in doc.sections()
        ]
    }
# Retrieval: first hit on doc_summary → then sections → then paragraphs

Hybrid (sliding-window) chunking

The pragmatic default: fixed-size windows that slide with overlap, snapping to natural breaks.

Use when
  • You want one strategy that "just works"
  • Answers often live on chunk boundaries
  • You're not sure → start here
Avoid when
  • Storage is critical (overlap = ~20% bloat)
  • You already have clean structure (use hierarchical)
Real-world
GitHub Copilot Chat over your repo

Code is chunked into ~800-token sliding windows snapped to function boundaries with 100-token overlap. A query about a helper that's used 200 lines away from where it's defined still retrieves both.

def hybrid_chunk(text, size=500, overlap=100):
    sentences = split_sentences(text)
    chunks, cur, count = [], [], 0
    for s in sentences:
        t = len(tokenize(s))
        if count + t > size:
            chunks.append(" ".join(cur))
            cur = cur[-overlap_sentences(cur, overlap):]   # keep tail
            count = sum(len(tokenize(x)) for x in cur)
        cur.append(s); count += t
    chunks.append(" ".join(cur))
    return chunks

Five Types of Simple RAG

Once you understand the base pipeline, "Simple RAG" splits into five flavors. Each keeps the vanilla retrieve-then-generate spine but tweaks one component — and that one tweak changes the use-case fit dramatically.

Naive (Vector-only) RAG

The textbook flavor. Embed everything, semantic search, stuff results into prompt. Done.

Best for
  • Conceptual / paraphrased queries
  • FAQs, chatbots, doc Q&A
  • Small-to-mid corpora (< 1M chunks)
  • "Just get me started" demos
Weak at
  • Exact-term queries (product SKUs, error codes)
  • Acronym / proper-noun lookups
  • Queries that share no vocabulary with the answer
Production example
Workspace-wide question answering (Notion-class)

Every page in the workspace is embedded into a vector index once. Ask "How do we handle PTO requests?" and the HR-handbook section surfaces even though the page actually talks about a "vacation time approval workflow" — that vocabulary-bridging behavior is exactly what semantic search buys you.

# Naive RAG — the 15-line version
chunks = load_and_chunk("docs/")
vectors = model.encode(chunks)
index.add(vectors, chunks)

def answer(query):
    qv = model.encode(query)
    top_k = index.search(qv, k=5)
    context = "\n".join(c.text for c in top_k)
    return llm(prompt(query, context))

Hybrid RAG (BM25 + Vector)

Combine keyword precision with semantic recall. Best of both worlds, fused by Reciprocal Rank Fusion.

Best for
  • Code search (function names, identifiers)
  • Legal & medical (exact terms matter)
  • Product catalogs (SKUs, model numbers)
  • Acronym-heavy domains
Weak at
  • Pure conversational chat (vector alone is enough)
  • Resource-constrained envs (2 indexes = 2× memory)
Production example
Sourcegraph Cody — code search across millions of repos

A query "undefined behavior in memcpy" needs both: BM25 to surface every file mentioning memcpy, and vector search to also find related concepts like "unaligned pointer copy". Hybrid lifts recall by 15-30% on technical queries.

# Reciprocal Rank Fusion (RRF) — the standard merge
def rrf(bm25_hits, vec_hits, k=60):
    scores = {}
    for rank, doc in enumerate(bm25_hits):
        scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank)
    for rank, doc in enumerate(vec_hits):
        scores[doc.id] = scores.get(doc.id, 0) + 1 / (k + rank)
    return sorted(scores.items(), key=lambda x: -x[1])

# OpenSearch supports it out of the box via hybrid query DSL

Hierarchical RAG

Index at multiple granularities. Search top-down. Stop when you have the right paragraph.

Best for
  • Very long documents (manuals, books, codebases)
  • Regulated industries needing precise citations
  • Multi-document QA across thousands of files
  • "Compare what doc A vs doc B says about X"
Weak at
  • Small flat corpora (overhead > benefit)
  • Documents without natural hierarchy
  • Tight ingestion budgets (summaries = extra LLM calls)
Production example
Legal-research assistants (Harvey-class)

Court opinions are organised in three nested layers — case, headnote, paragraph — and the retriever walks them top-down. A search for "fair use precedent in software copyright" first filters to roughly fifty plausible cases, then zooms into the individual opinion paragraphs, so the citation a lawyer ends up reading still has a verifiable provenance.

def hierarchical_retrieve(query):
    qv = model.encode(query)
    # Stage 1: find candidate docs
    docs = doc_index.search(qv, k=10)
    # Stage 2: find candidate sections within those docs
    sections = section_index.search(qv, k=5,
                          filter="doc_id IN " + ids(docs))
    # Stage 3: final paragraphs
    return para_index.search(qv, k=3,
                          filter="section_id IN " + ids(sections))

HyDE (Hypothetical Document Embeddings)

Short queries embed poorly. So: have the LLM imagine the answer first, then search with that.

Best for
  • Very short queries ("explain X", "what is Y?")
  • Zero-shot search on domain you weren't trained on
  • When a reranker is too expensive
  • Academic / long-form content lookup
Weak at
  • Latency-critical apps (extra LLM call = +500 ms)
  • Cost-constrained pipelines
  • Exact-fact lookups (hypothesis may hallucinate too far)
Production example
Conversational web answers (Perplexity-class)

When a user types a two-word fragment like "powerhouse cell", the raw query carries almost no signal. HyDE sidesteps that by asking the LLM to draft a paragraph-long candidate answer first, then embedding the draft and matching against the corpus — which is why such a sparse query still lands on real cellular-respiration papers.

def hyde_retrieve(query):
    # Step 1: LLM drafts a hypothetical answer
    draft = llm(f"Write a one-paragraph answer to: {query}")
    # Step 2: embed the DRAFT (not the query)
    qv = model.encode(draft)
    # Step 3: vector search as usual
    return index.search(qv, k=5)

answer = llm(prompt(query, hyde_retrieve(query)))
# the hypothesis is THROWN AWAY — only real chunks ground the final answer

Reference-based (Lightweight) RAG

The vector DB only knows where the text lives. Fetch the actual content from the authoritative source on demand.

Best for
  • Privacy-sensitive content (PHI, PII, classified)
  • Frequently-edited source documents
  • Per-user access control (row-level security)
  • Cost-sensitive storage (1 TB corpus → 100 GB vector DB)
Weak at
  • Read-heavy public corpora (extra fetch = latency)
  • Sources without stable offsets (live web pages)
  • Air-gapped vector DB deployments
Production example
Microsoft 365 Copilot — chat with your tenant's data

Indexing Outlook, OneDrive, Teams for billions of users → storing the original text in a vector DB would duplicate petabytes and create a giant access-control nightmare. Copilot stores only embeddings + Graph API references. At query time, the user's existing M365 permissions decide whether the referenced item is hydrated. Revoke a file? Vector still exists but yields no content.

# Index time: store ONLY pointer + vector
for chunk in chunks:
    vec = model.encode(chunk.text)
    index.add(vec, {
        "doc_id": chunk.doc_id,
        "offset": chunk.offset,
        "length": chunk.length,
    })   # note: NO 'text' field

# Query time: hydrate from source-of-truth, with ACL check
def retrieve(query, user):
    hits = index.search(model.encode(query), k=5)
    return [source.fetch(h.doc_id, h.offset, h.length, user)
            for h in hits if source.can_read(h.doc_id, user)]

Re-ranking — the quality control layer

Your vector search returned 20 "relevant" chunks. Half of them are noise. Pass the noisy 20 to your LLM and the answer will be confidently wrong. A re-ranker is the second pass that re-scores those candidates with a deeper model so only the truly relevant 3 reach the prompt — the difference between 70% and 95% answer accuracy.

The research-paper analogy

Writing a research paper, you first pull every source the library catalog returns for your topic. That's retrieval. Then you sit down and critically read each one — checking credibility, depth, fit — and pick the handful you'll actually cite. That second pass is re-ranking. The LLM only sees the chosen few.

The two-stage retrieval pattern — fast recall, then precise re-ranking

Why basic retrieval alone falls short

1
Shallow semantic match

Embeddings score by overall vector closeness — they miss subtle nuance ("not recommended" embeds close to "recommended").

2
Dimensionality collapse

Compressing meaning to 384–1536 floats causes collisions: unrelated concepts can land near each other in vector space.

3
Domain drift

Embeddings perform best on data resembling their training set. Out-of-domain corpora (legal, medical, code) tank recall.

4
"Lost in the middle"

LLM recall drops as you stuff more context. Top-20 unsorted > Top-3 reranked is almost never true.

The five reranker families

Each trades latency, cost, and quality differently. Click through:

BM25 — the lightweight lexical reranker

No ML model, no GPU — just term frequency, document length and inverse-doc-frequency. Often the first reranker to try.

Best for
  • Keyword-rich queries (SKUs, errors, names)
  • Cheap reranking of 100–1000 candidates
  • As the lexical leg of a hybrid pipeline
  • Resource-constrained / edge deployments
Weak at
  • Paraphrased queries (zero term overlap)
  • Conceptual / multi-hop reasoning
  • Long-form narrative ranking
Measured outcome
+20% precision on keyword-heavy medical search

Reranking the top-20 vector hits for "symptoms of diabetes" with BM25 surfaces concise, term-rich passages over verbose ones — boosting precision@3 by 20 percentage points in published medical RAG benchmarks.

from rank_bm25 import BM25Okapi

top20 = vector_index.search(query, k=20)        # Stage 1
tokens = [d.text.split() for d in top20]
bm25 = BM25Okapi(tokens)
scores = bm25.get_scores(query.split())
ranked = sorted(zip(top20, scores), key=lambda x:-x[1])
top3 = [d for d,_ in ranked[:3]]                # Stage 2 done

Cross-Encoder — the gold standard for relevance

A small transformer (e.g. ms-marco-MiniLM-L-6-v2) ingests query+doc together, attends across both, returns one relevance score per pair.

  • Why it wins: cross-attention lets the model spot which word in the query matches which word in the doc, including negations and modifiers.
  • Why it's slow: N candidates = N forward passes. Cannot be precomputed.
  • Sweet spot: rerank top 20–50 candidates — anything more becomes the latency bottleneck.
  • Open-source picks: bge-reranker-base / -large, ms-marco-MiniLM, mxbai-rerank.
Best for
  • Quality-critical RAG (legal, medical, research)
  • Reranking 20–50 vector hits
  • Self-hosted, no API dependency
Weak at
  • Real-time chat (adds 200–500 ms)
  • Reranking 1000+ candidates
  • CPU-only deployments at scale
Production example
Academic paper search — +40% citation precision

Reranking top-20 ArXiv embedding hits with bge-reranker-large identifies the deeply-relevant papers (not just topically similar) — lifting cited-paper precision by ~40% in published benchmarks.

from sentence_transformers import CrossEncoder

encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

top20 = vector_index.search(query, k=20)
pairs = [(query, d.text) for d in top20]
scores = encoder.predict(pairs)        # 20 forward passes
top3 = [d for _, d in
        sorted(zip(scores, top20), reverse=True)[:3]]

Multi-Vector / ColBERT — late interaction

Store one vector per token. At query time, sum the max-similarity from each query token to any document token. Token-level precision, vector-search speed.

  • Late interaction = query + doc encoded independently (cacheable), only the comparison happens online.
  • Storage cost: ~10–50× the bytes of a single-vector index. The price of token-level precision.
  • Modern variants: ColBERTv2 quantizes to ~30× less storage; PLAID makes search practical at billion-doc scale.
  • Why it matters: beats single-vector retrievers and rerankers on out-of-domain queries.
Best for
  • Mid-latency RAG (50–150 ms rerank budget)
  • Out-of-domain corpora where embeddings struggle
  • End-to-end retriever (no separate reranker)
Weak at
  • Storage-tight deployments
  • Very long documents (token count explodes)
  • Multilingual at scale (training data limits)
Production example
Vespa & Qdrant ColBERT mode for enterprise search

Vespa and Qdrant both ship native ColBERT support — enterprises use it as a single-stage retriever that matches cross-encoder rerank quality at < 80 ms p99 on 100M doc corpora.

from ragatouille import RAGPretrainedModel

rag = RAGPretrainedModel.from_pretrained("colbert-ir/colbertv2.0")
rag.index(documents)               # pre-computes token vectors

hits = rag.search(query, k=10)   # MaxSim late-interaction
# hits already reranked by token-level relevance

LLM-as-Judge — use the model itself to rank

Skip the dedicated reranker model. Have GPT-4 / Claude / a small fine-tuned LLM (RankT5, RankZephyr, RankGPT) judge relevance via prompting.

  • Zero training data needed. Pure prompt engineering on a model you already pay for.
  • Quality ceiling: a strong LLM judge often beats specialized rerankers on novel domains.
  • Cost ceiling: a single pairwise sweep over top-20 = ~190 LLM calls per query. Listwise cuts it to one call but blows the context window.
  • Fine-tuned variants (RankZephyr, RankGPT-4) compress this into a single ~7B model call with API-grade quality.
Best for
  • Domains with no labelled training data
  • Low-volume, high-stakes queries
  • When you need explanations for ranking
Weak at
  • High-QPS production traffic
  • Tight latency budgets (< 200 ms)
  • Cost-sensitive consumer apps
Practical use
Eval & offline-labelling pipelines

Most teams use LLM-as-judge to generate gold labels offline, then train a cheap cross-encoder on those labels for production. Best of both worlds: GPT-4 quality, MiniLM latency.

# Listwise reranking in a single LLM call
prompt = f"""Rank these documents by relevance to the query.
QUERY: {query}

DOCS:
[1] {docs[0]}
[2] {docs[1]}
[3] {docs[2]}
…

Return JSON array of IDs in ranked order, best first."""

ranked_ids = json.loads(llm(prompt))    # e.g. [4, 1, 7, 2, ...]
top3 = [docs[i-1] for i in ranked_ids[:3]]

API & lightweight rerankers — production defaults

Managed reranker APIs (Cohere, Jina, Voyage, Mixedbread) and the lightweight FlashRank library cover 80% of real RAG deployments.

  • Cohere Rerank v3.5 — the de-facto enterprise default. Multilingual, one HTTP call, integrates natively with LangChain's ContextualCompressionRetriever.
  • FlashRank — distilled rank-T5 in a 4 MB Python package. Runs on CPU at sub-50 ms for top-20 — ideal for chatbots and edge.
  • Jina & Voyage — long-context (8k+ tokens per doc) for technical / scientific corpora.
  • Mixedbread mxbai-rerank-v2 — open-source SOTA on the BEIR benchmark, self-hostable.
Best for
  • You want "just add reranking" with one line of code
  • Multilingual or domain-specific traffic
  • No GPU available locally (FlashRank is CPU-only)
Weak at
  • Air-gapped / on-prem only environments (API)
  • Extreme cost optimization at high QPS
Reproducible benchmark
~23% lower latency with a tighter, more on-point answer

A published walk-through on a several-hundred-chunk study corpus replaced a plain vector retriever with a Cohere-rerank wrapper, trimming the LLM-facing context from twenty passages down to three. End-to-end query time dropped from roughly 3.15 s to 2.42 s, and the generated answer was visibly tighter — fewer hedges, fewer off-topic asides.

from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import CohereRerank

base = vectorstore.as_retriever(search_kwargs={"k": 20})
compressor = CohereRerank(model="rerank-v3.5", top_n=3)
retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base,
)
docs = retriever.get_relevant_documents(query)   # 3 reranked docs

# --- or FlashRank for zero-API, sub-50ms reranking ---
from flashrank import Ranker, RerankRequest
ranker = Ranker(model_name="rank-T5-flan")
top3 = ranker.rerank(RerankRequest(query=query, passages=top20))[:3]

Reranker Latency (top-20) Quality Cost Setup Pick when…
BM25 ~5 ms good free trivial Keyword-heavy queries, edge devices
Cross-Encoder (MiniLM) ~150 ms excellent free needs GPU at scale Self-hosted high-quality RAG
ColBERT / Multi-vector ~50 ms excellent storage-heavy moderate End-to-end (no separate retriever needed)
LLM-as-Judge (GPT-4) ~2–6 s SOTA $$$ just prompt Eval / offline labelling / low QPS

The decision tree in one sentence
FlashRank for chat · Cohere for prod · Cross-encoder for self-hosted quality · LLM-as-judge for offline eval.

Almost every production RAG team starts with Cohere's managed API to prove value, then graduates to a self-hosted bge-reranker-base cross-encoder once cost/latency matters. Use LLM-as-judge to label a few thousand queries, then distill those labels into the cheap cross-encoder you actually serve.

Worked Example — a real query, end to end

One concrete trace beats ten diagrams. Below we walk a single user query "What organelle produces energy in human cells?" through every stage of a Naive RAG pipeline indexed over a biology textbook.

1

Load & chunk the textbook

The PDF is parsed by PyMuPDF, stripped of headers and figure captions, and split with sliding-window chunking (size = 512 tokens, overlap = 64).

Input: textbook.pdf · 412 pages · ~280k tokens Output: 617 chunks stored as {chunk_id, text, page, vector_pending} Sample chunk #142: "Mitochondria are membrane-bound organelles found in most eukaryotic cells. They generate the majority of the cell's supply of adenosine triphosphate (ATP), used as a source of chemical energy. The number of mitochondria varies widely by organism, tissue, and cell type…"
2

Embed every chunk (offline, one time)

Each chunk passes through all-MiniLM-L6-v2, producing a 384-dim float vector. Vectors + chunk text + page numbers are written to Qdrant.

Model: all-MiniLM-L6-v2 (80 MB, runs on CPU) Throughput: ~1,200 chunks/sec on a laptop Output: 617 × 384 = 236,928 floats → 924 KB index Chunk #142 vector (first 8 of 384 dims): [ 0.082, -0.341, 0.671, -0.124, 0.029, 0.488, -0.205, 0.137, … ]
3

User asks a question

Live request hits the API. The query is sent through the same embedding model.

Raw query: "What organelle produces energy in human cells?" Tokenized: ['what', 'organ', '##elle', 'produces', 'energy', 'in', 'human', 'cells', '?'] → 9 tokens Query vector (first 8 dims): [ 0.094, -0.328, 0.652, -0.103, 0.041, 0.502, -0.198, 0.144, … ] Latency: 28 ms (CPU embedding)
4

Top-K vector search

Qdrant's HNSW index returns the 5 closest chunks by cosine similarity. Notice the top hit shares zero words with the query — pure semantic match.

Search latency: 8 ms across 617 vectors (HNSW) Top-5 results (cosine similarity): 1. chunk #142 (sim=0.892) p. 87 — "Mitochondria are membrane-bound organelles…" 2. chunk #144 (sim=0.811) p. 88 — "ATP synthase complex pumps protons across…" 3. chunk #098 (sim=0.704) p. 61 — "Cellular respiration occurs in three stages…" 4. chunk #207 (sim=0.687) p. 126 — "Plant cells also contain chloroplasts which…" 5. chunk #143 (sim=0.643) p. 87 — "Cristae are the inward folds of the inner…"
5

Augment the prompt

The top chunks are stitched into a strict prompt template that grounds the LLM and demands citations.

Final prompt sent to GPT-4: You are a biology assistant. Answer ONLY from the context below. If the answer isn't in the context, say "I don't know". Cite the page number of each fact in parentheses. CONTEXT: [1] (p.87) Mitochondria are membrane-bound organelles found in most eukaryotic cells. They generate the majority of the cell's supply of adenosine triphosphate (ATP), used as a source of chemical energy… [2] (p.88) ATP synthase complex pumps protons across the inner membrane… [3] (p.61) Cellular respiration occurs in three stages: glycolysis, citric acid cycle, and the electron transport chain… QUESTION: What organelle produces energy in human cells? ANSWER: Total prompt size: ~720 tokens (well under the 8k window)
6

LLM generates the grounded answer

GPT-4 reads the context, ignores its priors, and returns a citation-rich answer in plain English.

Generation latency: 1.1 s Output tokens: 48 ▶ "Mitochondria are the organelles that produce energy in human cells. They generate ATP, the cell's main chemical energy currency, through cellular respiration (p. 61, p. 87). ATP synthase pumps protons across the inner mitochondrial membrane to drive ATP production (p. 88)." End-to-end query latency: 28 + 8 + 1100 ≈ 1.14 s Cost per query (GPT-4o): ~$0.0027 in + $0.0010 out = ~$0.004

Production Pitfalls

The pipeline looks tidy on a slide. In production, here's where it actually breaks.

Mismatched tokenizer/model between ingest & query

You embedded the corpus with bge-small but at query time you call ada-002. Vectors live in two unrelated spaces — recall collapses to ~5%.

Fix: pin the model name + version in config; re-embed everything if you change.

Forgetting to store the original text

Embeddings are not reversible. If your vector DB only holds vectors, you can't show the chunk to the LLM — only its ID.

Fix: always store {id, vector, text, source_uri, page} together — or use Reference-based RAG with stable offsets.

Chunk too big > embedding model's max

Most sentence embedding models silently truncate at 512 tokens. A 2000-token chunk becomes a 512-token chunk with no warning, and 75% of the content has zero influence on the vector.

Fix: check model.max_seq_length and size chunks ≤ that, with overlap to preserve context.

No "I don't know" path

When retrieval returns junk, the LLM happily makes something up. Worse: it sounds confident.

Fix: threshold on similarity score. If top-1 < 0.5, return "I couldn't find a reliable source." Bake into the prompt: "answer ONLY from context."

Trying to pass embeddings to the LLM

You can't. An LLM accepts tokens, not 384 floats. The vector is just an index key — you must look up the human-readable text and put that in the prompt.

Fix: retrieve top_k.text, not top_k.vector, into the prompt.

Ignoring stale or duplicated chunks

Re-ingesting the same wiki page nightly without dedupe → 30 near-identical chunks crowd the top-K. The LLM sees one fact six times, costs go up 6×, novelty drops to zero.

Fix: content-hash each chunk; upsert by hash. Diff against the previous run before reindexing.

No reranking when K is small

Vector search ranks by cosine, which is a coarse signal. A cross-encoder reranker (e.g. bge-reranker-base) on the top-20 can push your top-3 precision from 70% → 90%.

Fix: retrieve K=20 by vector, rerank to K=3 with a cross-encoder before sending to LLM.

Stuffing too much context into the prompt

More context ≠ better answers. LLMs suffer from "lost in the middle" — facts buried at prompt position 8/10 get ignored. Token cost goes up linearly; quality often drops.

Fix: keep K ≤ 5, put the most relevant chunk first, and trim each chunk to its most relevant ~200 tokens.
