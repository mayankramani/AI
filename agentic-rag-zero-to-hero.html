What is Agentic RAG?

Traditional RAG retrieves once and answers. Agentic RAG embeds autonomous agents into the pipeline — they plan, reflect, call tools, and collaborate to handle dynamic, multi-step reasoning.

A vanilla RAG system is a single shot: query → embed → retrieve → stuff into prompt → answer. It breaks down the moment a task needs multi-hop reasoning, real-time tool calls, iterative correction, or coordination between specialists. Agentic RAG replaces that single-shot pipeline with one or more agents that own the workflow end-to-end.

● Traditional RAG

One-shot retrieval + generation.

  • Static pipeline
  • No self-correction
  • Single index, single query
  • Limited multi-step reasoning

★ Agentic RAG

Agents orchestrate retrieval + reasoning.

  • Plans & rewrites queries
  • Reflects on its own output
  • Uses tools, web search, APIs
  • Multi-agent collaboration

▣ Agentic Document Workflows

End-to-end document automation.

  • Parses + structures docs
  • Maintains state across steps
  • Applies business rules
  • Produces actionable output

The Four Agentic Patterns

Every flavor of Agentic RAG is built from four primitives. Click each tab to walk through the loop, the steps, what makes it work — and the pseudo-code you can ship today.

Generate → Critique → if not good enough, loop back and refine.

🔁 Reflection — the inner critic

The agent grades its own output and refines until it clears an objective bar. Think of it as "draft → review → revise" baked into the model loop.

  1. 1 Draft. Generate the first attempt at an answer (or plan, or code).
  2. 2 Self-evaluate. A critic prompt (or a separate model) scores the draft against an explicit rubric — groundedness, completeness, format.
  3. 3 Decide. If score ≥ threshold → return. Otherwise capture the critique as targeted feedback.
  4. 4 Refine. Feed the feedback back into the generator. Loop with a hard cap (e.g. ≤ 3 iterations) to avoid runaway cost.
Use it when
  • You have a clear eval rubric
  • High-stakes / regulated output
  • Mistakes are expensive
Avoid when
  • Tight latency budget (< 1 s)
  • No good way to score quality
  • Tasks are deterministic
Real-world example
Self-RAG (arXiv 2310.11511)

Trains an LLM to emit special reflection tokens that gate retrieval and grade its own answer for groundedness — the model decides when to retrieve and whether its own response is actually supported by evidence.

# Reflection loop — pseudo-code def reflect_answer(question, max_iters=3, tau=0.85): draft = llm.generate(question) for i in range(max_iters): review = critic.score(draft, rubric) if review.score >= tau: return draft draft = llm.refine(draft, review.feedback) return draft # best-effort after cap
A single goal becomes an ordered, dependency-aware plan of sub-tasks.

🗺️ Planning — the strategist

The agent thinks before it acts. It breaks a fuzzy goal into a concrete, ordered plan — choosing what, when, and in what order.

  1. 1 Understand goal. Parse the user's intent, constraints, and success criteria.
  2. 2 Decompose. Emit an ordered list of sub-tasks (often as JSON) with dependencies.
  3. 3 Execute. Hand each sub-task to a retriever, a tool, or another agent.
  4. 4 Re-plan. If a step fails or surfaces new info, adjust the remaining plan.
  5. 5 Synthesize. Aggregate step outputs into a single coherent answer.
Use it when
  • Multi-step / long-horizon goals
  • Mixed tools or data sources
  • Order & dependencies matter
Avoid when
  • Single-shot lookups
  • Latency budget < 2 s
  • Tasks change every turn (use ReAct)
Real-world example
LangGraph Plan-and-Execute Agent

A planner LLM produces a JSON plan up front; an executor runs each step with tools and writes results back to shared state. If a step fails, a re-planner mutates the remaining plan. Used in production for research, data-pipeline, and multi-doc QA workflows.

# Plan-and-execute — pseudo-code def plan_and_run(goal): plan = planner.decompose(goal) # [step1, step2, ...] state = {} for step in plan: result = executor.run(step, state) state[step.id] = result if result.needs_replan: plan = planner.revise(plan, state) return synthesizer.merge(state)
The agent reasons, picks a tool, calls it, observes the result, then continues.

🛠️ Tool Use — extending reach

The agent escapes the limits of its training data by calling typed tools — search, vector DB, calculator, code interpreter, business APIs.

  1. 1 Reason. "What information do I still need?" The model writes a thought.
  2. 2 Pick. Choose the right tool from a typed registry (each with name, schema, description).
  3. 3 Call. Emit a structured action — usually JSON arguments matching the tool's schema.
  4. 4 Observe. Append the tool's output to the conversation as a new observation.
  5. 5 Iterate or finish. Loop until the agent has enough evidence to answer.
Use it when
  • Need fresh / private data
  • Math, code, or DB ops needed
  • Connecting to enterprise systems
Avoid when
  • Tools are flaky / unauthenticated
  • Sub-second latency mandatory
  • Answer is in the prompt already
Real-world example
Search-o1 (arXiv 2501.05366)

A search-enhanced reasoning model that interleaves chain-of-thought with live web search calls. The model decides — turn by turn — when its evidence is shaky and a new search would help, producing measurably better answers on long-tail technical questions.

# ReAct-style tool loop — pseudo-code def run_agent(question, tools, max_steps=8): history = [("user", question)] for _ in range(max_steps): step = llm.react(history, tools.schemas()) if step.action == "final": return step.answer result = tools[step.action].call(step.args) history.append(("tool", result)) raise RuntimeError("step budget exhausted")
Specialists collaborate via a shared blackboard; the coordinator delegates & merges.

🤝 Multi-Agent Collaboration — the team

Specialist agents divide the work. Each one has a narrow role and prompt; they share results through a coordinator or a blackboard.

  1. 1 Decompose by role. The coordinator decides who does what — retriever, reasoner, fact-checker, writer.
  2. 2 Parallel or sequential. Independent steps run concurrently; dependent ones run in order.
  3. 3 Share via blackboard. Each agent reads/writes a shared state (memory, scratchpad, or message bus).
  4. 4 Negotiate & vote. When two agents disagree, a critic or majority vote resolves the conflict.
  5. 5 Synthesize. The coordinator merges contributions into the final, attributable answer.
Use it when
  • Task spans multiple domains
  • You want specialization + audit trail
  • Steps can be parallelized
Avoid when
  • Coordination overhead > task cost
  • Single agent + tools already suffices
  • Strict deterministic output needed
Real-world example
LawGlance — legal research crew (CrewAI + LangChain + Chroma)

One agent retrieves statutes and case law from Chroma, a second performs targeted web search for recent rulings, and a third synthesizes a concise legal opinion with inline citations. Each agent's prompt and toolset is independent, making the system easy to extend with a new specialist (e.g. a tax-law expert) without touching the others.

# Multi-agent crew — pseudo-code shared = Blackboard() retriever = Agent(role="retriever", tools=[chroma, web_search]) analyst = Agent(role="analyst", tools=[reasoner]) writer = Agent(role="writer", tools=[formatter]) # sequential pipeline (could also be parallel) retriever.run(question, shared) analyst.run(shared) return writer.run(shared)

Agentic Workflow Patterns

Five proven control-flow shapes for assembling LLM-powered systems (Anthropic Research & LangGraph). Click each tab to explore.

1 · Prompt Chaining — Accuracy Through Sequential Steps

Decompose a task into ordered steps; each step refines the output of the previous one. Trades latency for accuracy.

When to use

Tasks with fixed, well-known sub-steps where each output cleanly feeds the next.

Example

Generate marketing copy in English → verify brand voice → translate to French while preserving nuance.

Real-world use case
Document-creation pipeline (Anthropic cookbook)

An LLM first generates an outline; a second call validates the outline meets all the required headings (gate); a third call expands it into the full document — measurably higher quality than asking for the document in one shot, at the cost of 3× the calls.

Tradeoffs
LatencyHigh
CostMed
AccuracyHigh
# Prompt chaining — pseudo-code outline = llm.call("Draft an outline for: " + topic) if not validator.passes(outline): raise GateFailed(outline) draft = llm.call("Expand this outline: " + outline) final = llm.call("Polish for tone & brand voice: " + draft) return final

2 · Routing — Send the Query to the Right Specialist

A classifier inspects the input and dispatches it to a specialized prompt, model, or sub-pipeline.

When to use

Different input categories deserve distinct handling strategies or models of different cost / capability.

Example

Customer-service queries routed to tech-support / refunds / general; cheap queries go to a small model, complex ones to GPT-class models.

Real-world use case
Tiered model serving — cost optimization

A small classifier inspects each query; ~70% are simple FAQs routed to a 7B local model at near-zero cost, ~25% go to a mid-tier model, and only ~5% complex queries reach GPT-class models. Companies routinely report 60-80% inference-cost reduction with no measurable quality loss.

Tradeoffs
LatencyLow
CostLow
AccuracyMed-High
# Routing — pseudo-code def route(query): label = classifier.predict(query) # cheap, fast if label == "tech_support": return support_agent.handle(query) elif label == "refund": return refund_agent.handle(query) elif label == "complex": return big_model.handle(query) return small_model.handle(query)

3 · Parallelization — Speed & Confidence via Concurrency

Run independent sub-tasks at the same time (sectioning), or generate multiple candidate answers and vote (voting).

When to use

Sub-tasks are independent (reduce latency) or multiple opinions raise confidence (voting).

Example

Three models concurrently scan code for security issues; majority vote determines whether to flag.

Real-world use case
Content moderation with voting (Anthropic)

Three independent prompts each judge whether a piece of user content violates policy. The system blocks only when ≥ 2 of 3 agree — drastically reducing both false positives and false negatives versus a single judgment, with the same wall-clock latency as one call.

Tradeoffs
LatencyLow
CostHigh
AccuracyHigh
# Parallel voting — pseudo-code (asyncio) async def judge(content): verdicts = await asyncio.gather( model_a.judge(content), model_b.judge(content), model_c.judge(content), ) return majority(verdicts) # 2-of-3 wins

4 · Orchestrator-Workers — Dynamic Task Delegation

A central orchestrator decides at runtime how to break the task down, delegates to specialist workers, then synthesizes.

When to use

Sub-tasks are not known up front; the orchestrator decides on the fly based on input complexity.

Example

A coding agent: orchestrator inspects the diff request, then spawns workers to edit only the relevant files and run only the affected tests.

Real-world use case
Multi-source research agent

For a question like "How are competitors pricing tractor part X?", the orchestrator dynamically spawns workers: one searches news, one queries the catalog DB, one scrapes competitor sites, one summarizes earnings transcripts — then merges everything into a single comparison report. The shape of the plan depends entirely on what the user asked.

Tradeoffs
LatencyMed
CostMed-High
AccuracyHigh
# Orchestrator-workers — pseudo-code def handle(task): plan = orchestrator.decide(task) # runtime decomposition results = [] for sub in plan.subtasks: worker = workers.pick(sub.kind) results.append(worker.run(sub)) return orchestrator.synthesize(task, results)

5 · Evaluator-Optimizer — Iterative Refinement

One model generates; a second model evaluates and gives critique; the generator refines. Loop until quality criteria pass.

When to use

Quality clearly improves with iteration AND you have an explicit evaluation rubric.

Example

Literary translation: generator drafts, evaluator scores fidelity + style; loop 2-3× until both metrics clear thresholds.

Real-world use case
Iterative literary translation

A generator drafts a chapter translation; an evaluator scores it on fidelity, fluency, and register, and writes a precise critique ("line 14 loses the metaphor"). Two to three loops typically clear the threshold — published benchmarks show this beats single-shot translation on long-form prose.

Tradeoffs
LatencyHigh
CostHigh
AccuracyV. High
# Evaluator-optimizer — pseudo-code def refine(prompt, max_iters=4, tau=0.9): draft = generator.make(prompt) for _ in range(max_iters): eval_ = evaluator.score(draft) if eval_.score >= tau: return draft draft = generator.make(prompt, feedback=eval_.critique) return draft

The Memory Layer

Every agentic pattern above silently assumes a memory system underneath. An LLM is stateless; an agent behaves as if it remembers. Here's the stack that makes the illusion real — from a single context window to versioned, multi-agent persistent stores.

⚠️ Stateless LLM — every call is a goldfish

The model receives text, returns text, and retains nothing. No internal store is updated between calls.

  • Double-books the calendar — doesn't know what's scheduled
  • Resets tone & style every session — no user preferences
  • Repeats failed search queries — no memory of dead ends
  • Forgets mid-task discoveries — "file missing" lost on next call

✨ Memory-Augmented Agent — behaves as if it remembers

A stack of in-context + external stores gives the stateless model the illusion of persistent, queryable knowledge.

  • Reads & writes a calendar entity store before every booking
  • Personalizes from a per-user preference profile
  • Logs failed queries to episodic memory — won't retry blindly
  • Persists mid-task facts to a scratchpad / blackboard

The 4 questions that define the memory problem

1
What happened before?

Calendar booking, ticket history, prior decisions — ignored = double-booking, contradictions.

2
What does this user want?

Tone, format, preferences, language. Without it, the agent feels generic on every turn.

3
What has the agent tried?

Failed queries, dead-end branches, exceptions. Without it, the same wall gets hit twice.

4
What facts accumulated?

Mid-task discoveries (file missing, API down) need to persist into the next step.

The Memory Stack

Four memory tiers, each with a different scope, fidelity, and timescale. Production agents almost always combine all four.

In-Context ~minutes · 128K–1M tokens
Episodic hours–days · event log
Semantic weeks–∞ · entity store
Procedural months–∞ · skill / playbook

The Memory Lifecycle — from experience to action

✍️ Writing to Memory — selective by design

An agent that writes everything creates noise faster than signal. Memory needs a filter.

  • End-of-session summarization — a dedicated step extracts salient facts, decisions, and outcomes into compact records.
  • Event-triggered writes — fire on user corrections, explicit preferences, task completions, and error conditions.
  • Deduplication — check vector similarity against existing memories before inserting; merge near-duplicates.
  • Never store — raw transcripts at scale, intermediate CoT that won't affect future behavior, redundant duplicates.
Trigger on
  • "I prefer X" / "call me Y"
  • User correction of agent
  • Task completion / outcome
  • Error + recovery strategy
Drop
  • Raw token-by-token traces
  • Internal reasoning scratch
  • Duplicate of recent record
  • Sensitive PII (unless required)
# Selective end-of-session write — pseudo-code def consolidate(session, store): summary = summarizer.extract(session, schema=[ "preferences", "decisions", "outcomes", "errors"]) for rec in summary.records: near = store.search(rec.text, k=3) if near and near[0].score > 0.92: store.merge(near[0].id, rec) # dedupe else: store.insert(rec, ts=now(), agent=session.agent_id)

🔍 Retrieving from Memory — hybrid by default

Vector search is fuzzy. Structured query is brittle. Run both in parallel and merge.

  • Vector similarity — embed query, top-K from HNSW/IVF index. Fast, approximate, great for unstructured memories.
  • Structured query — SQL or key-value lookup by user, timestamp, entity. Precise but can't handle semantic drift.
  • Hybrid retrieval — the canonical case: "billing issues from the last 30 days for this user" needs both signals.
  • Re-rank — use RRF (Reciprocal Rank Fusion) or a cross-encoder to merge the two ranked lists into one.
Production tip
Always carry a structured filter

Most "the agent retrieved someone else's data" bugs come from pure-vector search missing a user_id filter. Apply structured constraints before the vector search (pre-filter) whenever your store supports it.

# Hybrid retrieve — pseudo-code def hybrid_retrieve(query, user_id, k=10): sem = vector_db.search( embed(query), k=k, filter={"user_id": user_id, "age_days": "<30"}) stru = sql.query( "SELECT * FROM tickets WHERE user_id=? " "AND ts > now()-interval '30 day'", user_id) return rrf_merge(sem, stru, k=k)

⏳ Decay & Versioning — fighting stale facts

A confidently-surfaced outdated memory is worse than no memory at all.

  • Temporal decay — multiply each memory's relevance score by exp(-λ · age) at retrieval time.
  • Versioned entity records — never overwrite; insert a new version with a timestamp and mark prior as superseded.
  • TTL on volatile facts — deprecate API endpoints, prices, job titles automatically after N days.
  • Trust gradient — facts confirmed by recent user correction outrank older inferences.
Failure mode
The confident-stale answer

An agent that learned a user was "Engineer" 18 months ago and never re-checked will keep routing them to engineering content even after they're promoted to "Tech Lead". Versioning + decay is what prevents this silent drift.

# Decay-weighted retrieval + versioned upsert def score(mem, query_emb, lam=0.01): sim = cosine(mem.emb, query_emb) age = (now() - mem.ts).days return sim * exp(-lam * age) # temporal decay def upsert_entity(store, entity_id, attr, value): prev = store.current(entity_id, attr) if prev: store.mark_superseded(prev.version_id, ts=now()) store.insert_version(entity_id, attr, value, ts=now())

🔗 Multi-Agent Memory — the consistency triangle

When a coordinator + N specialists all read & write the same store, you pick a tradeoff — there's no free option.

  • Central store + locks — one source of truth, serialized writes via locks or optimistic concurrency. Simple to reason about; throughput bottlenecks under contention.
  • Per-agent namespaces — each agent writes to its own space, reads across spaces. No conflicts but no shared truth either; duplication grows.
  • Append-only event log — every change is an event; readers fold events into the view they need. Massive write throughput; conflicts resolved at read time (CRDT-style).
  • Hybrid in practice — most production systems use namespaces for working state + central store for canonical entities + append log for audit.
Real-world example
Anthropic's research subagents

The Claude research workflow spawns 3-10 parallel subagents that each maintain their own scratchpad (namespace), then write distilled findings back to a coordinator-owned shared memory (central, optimistic) which the lead agent reads to compose the final report.

# Optimistic concurrency on shared entity store def safe_update(store, entity_id, mutate_fn): for _ in range(5): # retry budget record, version = store.read(entity_id) new = mutate_fn(record) if store.cas(entity_id, version, new): # compare-and-swap return new raise ConflictError(entity_id)

📊 Evaluation — catching silent failures

Memory systems fail quietly: the agent retrieves something wrong, reasons from it, produces a plausible wrong answer. Instrument from day one.

  • Build a golden set — a few hundred (query, relevant_memory_ids) pairs your team agrees on. The seed of every memory eval.
  • Run on every retriever change — embedding model swap, chunking tweak, re-rank weight — all silently break recall.
  • Track staleness in production — sample retrievals, ask a judge model "is this fact still true today?" — cheap and surprisingly effective.
  • Faithfulness != quality — the agent can ignore retrieved memory and still sound right. Measure whether memory actually shows up in the answer.
Recall
Did we surface the relevant memory at all?
|retrieved ∩ relevant| / |relevant|
Precision
How much noise rode along?
|retrieved ∩ relevant| / |retrieved|
Faithfulness
Did the answer actually use retrieved memory?
judge(answer, retrieved)
Staleness
How often is the surfaced fact outdated?
|stale| / |retrieved|
# Nightly memory regression eval — pseudo-code def eval_memory(golden, retriever): metrics = {"recall": [], "precision": [], "stale": []} for q, relevant in golden: got = retriever.search(q, k=10) ids = {m.id for m in got} metrics["recall"].append(len(ids & relevant) / len(relevant)) metrics["precision"].append(len(ids & relevant) / len(ids)) metrics["stale"].append(staleness_judge(got)) return {k: mean(v) for k, v in metrics.items()}

Taxonomy of Agentic RAG Systems

Seven architectural archetypes — pick the simplest one that fits your task. Click each row to expand.

01

Single-Agent RAG — Router-based, the simplest agentic upgrade

Idea: One agent owns the entire flow — choose retriever, query it, generate answer.

Workflow: Query → agent classifies → agent picks tool (vector DB or web search) → retrieves → answers.

Pros
  • Easy to build & debug
  • Low coordination overhead
  • Cheap inference
Cons
  • Doesn't scale to many tools
  • Weak on multi-step reasoning
Best for
  • FAQ chatbots
  • Doc Q&A over one corpus
  • Internal knowledge-base search
Real-world use case
IBM Granite-3-8B knowledge assistant

A LangChain + ChromaDB + Watsonx.ai stack where a single agent routes between a vector index of corporate docs and a web-search tool, then synthesizes an answer. Documented in the AgenticRAG-Survey notebook collection.

02

Multi-Agent RAG — specialists collaborating

Idea: A team of agents, each with a narrow job, working on different parts of the same task.

Workflow: Coordinator splits the task → retriever-agent pulls docs → reasoning-agent analyses → synthesis-agent writes answer → results merged.

Pros
  • Modular & extensible
  • Each agent stays focused
  • Mix specialized tools
Cons
  • Coordination cost grows fast
  • Risk of redundant calls
Best for
  • Multi-domain knowledge
  • Cross-source synthesis
  • NL2SQL + RAG combos
Real-world use case — AgentFlow
Planner · Executor · Verifier · Generator

Four trainable modules coordinate through an evolving memory and integrated tools (python_coder, google_search, wikipedia_search). The planner is optimized with Flow-GRPO to handle long-horizon, multi-turn reasoning with sparse rewards — a strong example of multi-agent specialization beating a single big model.

03

Hierarchical Agentic RAG — manager + workers

Idea: A top-level manager agent delegates to lower-level specialist agents and integrates their results.

Workflow: Manager plans → delegates each sub-task to the right worker → workers report back → manager refines & ships.

Pros
  • Scales to large workflows
  • Clear separation of concerns
  • Easy to add new specialists
Cons
  • Manager can bottleneck
  • Needs robust orchestration
Best for
  • Enterprise research assistants
  • Multi-tool QA
  • Long-running automation
Real-world use case
crewAI hierarchical research crew

A manager agent coordinates a WeaviateTool (internal docs), an ExaSearchTool (web), and a Groq fast-inference summarizer to answer complex business questions — used for QA pipelines and workflow automation.

04

Corrective Agentic RAG (CRAG) — self-grading retrieval

Idea: A critic module grades each retrieved chunk; irrelevant ones are dropped and the query is rewritten or extended with a web search until evidence is good enough.

Workflow: Retrieve → grade → if weak: rewrite query / web-search → re-retrieve → answer when grounded.

Pros
  • Reduces hallucinations
  • Improves grounding
  • Self-recovers from bad retrieval
Cons
  • Extra LLM calls = cost + latency
  • Needs anti-loop guard
Best for
  • High-stakes Q&A
  • Medical / legal / compliance
  • Open-domain answers
Real-world use case
LangGraph CRAG tutorial

Retrieves docs from a Chroma index, runs a grader on each, and on failure rewrites the query and falls back to Tavily web search. Pattern is published in Corrective Retrieval-Augmented Generation (arXiv 2401.15884) and ships as a runnable LangGraph example.

05

Adaptive Agentic RAG — picks the strategy per query

Idea: Inspect each query's complexity and dynamically choose no-retrieval, single-step retrieval or multi-step retrieval.

Workflow: Classify query complexity → route to the cheapest strategy that can answer it correctly.

Pros
  • Cost-aware
  • Low latency for simple Qs
  • Deep reasoning when needed
Cons
  • Classifier accuracy is critical
  • Harder to evaluate end-to-end
Best for
  • Customer support at scale
  • Mixed-difficulty workloads
Real-world use case
Adaptive-RAG (arXiv 2403.14403)

Trains a small classifier that labels each question as simple / moderate / complex and dispatches to the matching pipeline. In published benchmarks it matches multi-step accuracy at a fraction of the cost on easy queries — useful for high-traffic assistants.

06

Graph-Based Agentic RAG — relational reasoning

Idea: Augment retrieval with a knowledge graph; agents traverse relations for multi-hop reasoning.

Workflow: Extract entities → expand related sub-graph → fuse with unstructured chunks → reason → critic validates.

Pros
  • Excels at multi-hop questions
  • Combines structured + text data
  • Higher relevance precision
Cons
  • Needs a curated KG
  • More infra (graph DB)
Best for
  • Healthcare (disease ↔ symptom)
  • Drug discovery
  • Compliance & org charts
Real-world use case
GeAR — Graph-Enhanced Agent for RAG (arXiv 2412.18431)

Expands the query into a relevant sub-graph (e.g., a patient's symptoms → linked diseases → relevant trials) and uses agents to multi-hop reason across nodes — outperforms flat vector retrieval on questions whose answer depends on a chain of relationships.

07

Agentic Document Workflows (ADW) — end-to-end document automation

Idea: Goes beyond Q&A — agents parse, reason about, and act on entire documents (invoices, contracts, claims).

Workflow: Parse & structure doc → maintain workflow state → retrieve references → apply business rules → emit structured output (report, payment recommendation, etc.).

Pros
  • Replaces manual ops work
  • State across multi-step flow
  • Domain-specific intelligence
Cons
  • Heavier infra
  • Needs domain rule encoding
Best for
  • Contract review
  • Invoice processing
  • Insurance claims
Real-world use case
LlamaCloud Invoice Payments Workflow

Parses an invoice (vendor, line items, payment terms), retrieves the matching vendor contract, validates terms, then generates a payment-recommendation report — flagging early-payment discount opportunities. The same pattern powers LlamaCloud's Contract Review and Auto-Insurance-Claims demos.

Worked Example: An ADW for Invoice Payments

Follow a real Agentic Document Workflow (LlamaCloud pattern) handling a single supplier invoice end-to-end.

Scenario: An accounts-payable team receives a PDF invoice from a tier-1 supplier. Instead of a clerk doing 6 manual steps, an ADW runs the same workflow autonomously and ships a payment recommendation with citations in under a minute.

1
Parse & structure the invoice

The document parser agent extracts vendor name, invoice number, line items, totals, payment terms (Net 30, 2/10 discount) and PO reference.

tool: LlamaParse / Azure Document Intelligence
2
Retrieve the matching contract

The retrieval agent queries the contract index for that vendor + PO and pulls the master service agreement, pricing schedule, and SLA.

tool: Redis Vector Store + LlamaIndex
3
Verify terms (CRAG-style)

The verifier agent compares invoice line items to contract pricing. If a unit price exceeds the contracted tier, it grades the evidence as insufficient and re-queries amendments & addenda before deciding.

pattern: corrective RAG + reflection
4
Multi-hop business-rule reasoning

The policy agent checks: budget allocation, approver delegation matrix, fraud-signal API (duplicate invoice number, abnormal amount), and tax jurisdiction rules.

tools: ERP API · Tax API · Fraud API
5
Identify optimization opportunity

The planner agent notices the invoice qualifies for a 2% early-payment discount (2/10 Net 30) and computes ROI vs. cash-flow forecast.

pattern: planning + tool use
6
Generate actionable output

The writer agent emits a structured payment-recommendation report: amount, payment date, approver, savings, risk flags, and inline citations to the contract clauses used.

output: JSON + human-readable PDF
7
Human-in-the-loop sign-off

The recommendation is routed to the right approver in the AP system. The agent stays "warm" to answer follow-up questions about its reasoning trace.

pattern: human-AI collaboration

The defining unit of agentic RAG is not the model itself but the control loop wrapped around it. What used to be a prompt-engineering exercise turns into a workflow-engineering one. — Synthesis informed by the Singh et al. survey (arXiv:2501.09136, 2025)
